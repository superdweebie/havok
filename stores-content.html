Overview

Dojo provides a superb data store api and a very solid selection of data store tools.

Stores are data repositories. They can be thought of as mini databases which store a single table, or collection of data.

Stores provide a consistent interface for supplying data to all parts of your application. Stores may be locally created in memory, backed by html5 local storage, a json rest api, or some other source.

Manager

The store manager is a container for managing data stores

The store manager is a central repository for your app's data stores. Stores can be added to the manager in many different ways, and then used throughout your application.

Adding a Store

Stores may be added to the store manager directly. Eg:

  1. require(['havok/store/manager', 'dojo/store/Memory'], function(storeManager, Memory){
  2.     storeManager.add('mystore', new Memory);
  3. }

Stores may also be added through markup and the shared di config. See below for details.

Getting a store

To get a store, pass the store name to the get function. Eg:

  1. require(['havok/store/manager'], function(storeManager){
  2.     storeManager.get('mystore').then(function(mystore){
  3.         //do something with mystore
  4.     })
  5. }

AMD Plugin

havok/store/manager is also an AMD plugin, allowing you to retrieve a store with the following shorthand:

  1. require(['havok/store/manager!mystore'], function(mystore){
  2.     //do something with mystore
  3. }

Datalist

There are many different ways to add a store to the manager, depending on your needs.

Stores can be added using markup. If you place a datalist anywhere in your document with it's id attribute set, then havok/store/Datalist will be used to turn that datalist into a store. Eg:

  1. <datalist id="myDatalistStore">
  2.     <option data-id="home">home</option>
  3.     <option data-id="navs">#navs anchor</option>
  4.     <option data-id="disabled">disabled pill</option>
  5. </datalist>
  6. <script>
  7. require(['havok/store/manager!myDatalistStore'], function(myDatalistStore){
  8.     //do something with myDatalistStore
  9. }
  10. </script>

Di Stores

Stores may be added using the shared di config. To do so, add keys to the stores object. Eg:

  1. di: {
  2.     'stores': {
  3.         gets: {
  4.             store1: 'my/store/one',
  5.             store2: 'my/store/two'
  6.         },
  7.         proxies: {
  8.             store3: {
  9.                 base: 'my/store/three',
  10.                 proxyMethods: [
  11.                     'get',
  12.                     'query'
  13.                 ]
  14.             }
  15.         }
  16.     }
  17. }

Adding stores via di is particularly handy for two reasons.

Firstly, it is very easy to override a store, and inject an alternative data set. For json rest stores, it can be convienient to inject the target.

Secondly, by using proxies, the store will be lazy loaded. That is, it will only be fetched on it's first use.

Throttle

havok/store/Throttle is a store wrapper that will throttle activity on a store. It is useful when using a store such as dojo/store/JsonRest to control the volume of server requests.

Internally throttle has a queue of requests. Each time a query, get, put, add, or remove is made against the store, the request is added to the queue. Items in the queue are processed at a set interval. Items in the queue are given a ttl (time to live). If an item has gone stale before it is processed, then the request is dropped.

Throttle takes three arguments:

  • store the store to be wrapped
  • throttle the minimum time in milliseconds between activity on the store. Defaults to 100.
  • ttl the time to live of each request.
  1. require(['havok/store/Throttle', 'dojo/store/JsonRest'], function(storeManager, JsonRest){
  2.     var store = new Throttle(
  3.         new JsonRest({target: 'http://myserver.com'}), //store to throttle
  4.         300, //JsonRest store will not recieve requests faster than one every 300 milliseconds
  5.         1000 //if any request is unprocessed for 1000 milliseconds, then it will be dropped
  6.     );
  7. }

If you need to detect success or failture of a request, use the returned promise:

  1. require(['havok/store/Throttle', 'dojo/store/JsonRest'], function(storeManager, JsonRest){
  2.     var store = new Throttle(new JsonRest({target: 'http://myserver.com'}), 300, 1000);
  3.  
  4.     store.query({name: 'Bob}).then(
  5.         function(data){
  6.             //request succeeded. Do something with data
  7.         },
  8.         function(err){
  9.             //ttl expired. Request dropped. Do something with err
  10.         }
  11.     );
  12. }

Query Cache

havok/store/QueryCache is a store wrapper that extends havok/store/Cache to also cache query results.

Eg:

  1. require(['havok/store/QueryCache', 'dojo/store/JsonRest', 'dojo/storeMemory'], function(storeManager, JsonRest, Memory){
  2.     var store = new QueryCache(
  3.         new JsonRest({target: 'http://myserver.com'}), //master store
  4.         new Memory //slave store
  5.     );
  6. }

By default the query cache will be cleared each time a put, add, or remove is made against the store.

The query cache can be cleared manually with the clearQueryCache function.

Widgets with Stores

Many of the havok widgets can optionally use the havok/widget/_StoreMixin to manage lists of items. Stores allow great flexibility. Items may be fetched via ajax, filtered, sorted, and reused throughout your application.

Nav

  • home
  • #navs anchor
  • disabled pill

  2.     <option>home</option>
  3.     <option href="#stores">#navs anchor</option>
  4.     <option disabled>disabled pill</option>
  5. </datalist>
  6. <w-nav-pill store="navStore1"></w-nav-pill>
